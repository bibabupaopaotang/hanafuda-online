# Hachi-Hachi 模块 — 游戏设计文档

> 校验日期：2026-04-27
> 依赖：`rule-engine.md`、`game-state.md`、`scoring.md`

## 概述

- **系统名称**：Hachi-Hachi 模块 (Hachi-Hachi Game Module)
- **所属层级**：Feature
- **优先级**：P2（MVP 之后实现）
- **设计目标**：实现 Hachi-Hachi（八八花札）三人对战模式的完整游戏流程。

## 模式基本信息

| 项目 | 值 |
|------|----|
| 人数 | 3 人 |
| 牌数 | 48 张 |
| 目标分 | 30 分（可配置） |
| Koi 机制 | **无** |
| 结算时机 | 山札耗尽后统一结算 |

## 与 Koi-Koi 的核心差异

| 项目 | Koi-Koi（两人） | Hachi-Hachi（三人） |
|------|-----------------|---------------------|
| 人数 | 2 | 3 |
| Koi | 有 | **无** |
| 结算时机 | 任一玩家凑出役即可喊停 | 山札耗尽后统一结算 |
| 得分累积 | 先到 7 分胜 | 全程累加，先到 30 分胜 |
| 发牌 | 手 8 + 场 8 + 山 24 | 手 8×3 + 场 8 + 山 16 |
| 特殊役 | 手四 | 手四、場四、カ斯役 |

## 发牌流程

```
1. 亲（庄家）先拿 9 张手牌
2. 亲选择 1 张弃至场中（使手牌变为 8 张）
3. 子 B 拿 8 张手牌
4. 子 C 拿 8 张手牌
5. 场牌：8 张
6. 山札：剩余牌（48 - 24 - 8 = 16 张）
```

## 回合流程

```
亲方出牌 → 匹配判定 → 翻山札 → 匹配判定
    │
    ▼
下一位玩家（按顺时针）→ 同样流程
    │
    ▼
山札耗尽 → 三方统一结算
```

**注意**：三模式**没有役判定中途结算**。每局打完整副牌（山札耗尽），然后结算各方的役分。

## 三人模式役表

| 役名 | 条件 | 点数 |
|------|------|------|
| 四光 | 4 张光牌 | 10 点 |
| 五光 | 5 张光牌 | 15 点 |
| 雨四光 | 4 张光牌（含小野道风） | 8 点 |
| 三光 | 3 张光牌（不含小野道风） | 6 点 |
| 赤短 | 3 张红色短册 | 5 点 |
| 青短 | 3 张青色短册 | 5 点 |
| 猪鹿蝶 | 猪(7月) + 鹿(10月) + 蝶(6月) | 5 点 |
| 手四 | 手牌含同一月 4 张 | 4 点 |
| 場四 | 场札全为同一月 | 8 点 |
| カ斯20枚 | 累计收集 20 张カ斯 | 10 点 |
| カ斯10枚 | 累计收集 10 张カ斯 | 5 点 |
| 呑み | 3月幕 + 8月月 + 9月杯 | 4 点 |
| 花见酒 | 3月幕 + 9月杯 | 3 点 |
| 月见酒 | 8月月 + 9月杯 | 3 点 |

## 特殊开局

| 情况 | 处理 |
|------|------|
| 场中四张同月 | 流局重发 |
| 手牌四张同月（手四） | 该玩家得 4 点，本局继续打完 |

## 结算流程

```
山札耗尽
    │
    ▼
三方各自计算役分
    │
    ▼
累计到总分
    │
    ▼
检查是否有人达到 30 分
    │
    ├─ 是 → 游戏结束，最高分者胜
    └─ 否 → 开始下一局
```

## 数据结构差异

```typescript
interface HachiHachiState extends GameState {
  mode: 'hachi_hachi';
  players: PlayerState[];     // 3 个玩家
  currentPlayerIndex: number; // 0, 1, 2
  discardPile: number[];      // 亲弃牌（用于记录）
  // 无 koiKoiCount/koiKoiCaller 字段
}
```

## 客户端交互

与 Koi-Koi 类似，但：
- 显示 3 个玩家的位置
- 不显示 Koi 决策弹窗
- 局中不显示役分（仅局末结算时显示）
- 显示当前山札剩余数量
